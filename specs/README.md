# Specs — Spec-Driven Development

Every feature goes through 3 artifacts before being implemented:

```
specs/NNN-slug/
├── spec.md     # WHAT + WHY (product language, no tech)
├── plan.md     # HOW technical (structure, schemas, routes, tests)
└── tasks.md    # Executable checklist task-by-task
```

## Navigation

- [`INDEX.md`](INDEX.md) — overview by status (in-progress, approved, draft, done, archived). Regenerated by `/specs-index`.
- [`archive/`](archive/) — specs done > 30 days or abandoned, grouped by quarter. See [archive/README.md](archive/README.md).

## Flow

```
/specify "description in natural language"
   ↓ (human reviews spec.md, answers [NEEDS CLARIFICATION])
/plan
   ↓ (human reviews plan.md)
/tasks
   ↓ (human reviews granularity)
/implement   (executes ONE task; run N times until done)
   ↓ (after Status: done for > 30 days)
/specs-archive [NNN-slug]   →   moves to archive/YYYY-Qn/
```

Each `/` is a slash command in `.claude/commands/`. See also the `new-feature` skill which leads the full cycle.

### Other maintenance commands

- `/specs-index` — regenerates `INDEX.md` by reading the `Status:` of each spec.
- `/specs-archive [NNN-slug]` — archives a done/abandoned spec.

## Why SDD?

- **Spec separates "what" from "how"** — product decision-makers aren't at the mercy of implementers.
- **Reviewed plan prevents rework** — design issues are found before code is written.
- **Small tasks enable AI** — agents make fewer mistakes on short, well-defined scope.
- **Audit trail** — every major change has a history of decisions.

## Convention

- NNN is zero-padded (001, 042, 100…). Next number = `max(NNN existing in specs/ AND archive/) + 1`.
- Slug in short kebab-case (3-5 words).
- Status at the top of `spec.md`: `draft | approved | in-progress | done | abandoned`.
- After creating/updating a spec, run `/specs-index` to keep `INDEX.md` in sync.

## When NOT to use SDD

- Copy/text change.
- Small bug fix (< 30min).
- Refactor with no behavior change.
- Isolated visual adjustment.

For these cases, do it directly and describe in the commit.
